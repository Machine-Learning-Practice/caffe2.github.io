109 // Note [User node does not uniquely identify use]
110 // ~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~~
111 // A while back, we wrote some code manipulating uses that looked like this:
112 //
113 // for (auto& use : used_val->uses_) {
114 // if (use.user == this_node) {
115 // use.offset += 1;
116 // break;
117 // }
118 // }
119 //
120 // This code is trying to find a particular use (our node's use) to update it.
121 // However, it's wrong: there may be *multiple* uses of a value %x in a node,
122 // as might be the case in this IR:
123 //
124 // %y = Add %x %x
125 //
126 // In this case, there are two uses of %x whose user is the node 'Add %x %x'.
127 // So, "use induced by this node" is not a well-formed concept.
128 //
129 // If you are looking for "use induced by an input", it's best to use
130 // findUseForInput() to get it.

453  // Each Node can have a list of subblocks. These are used to define structured
454  // nested control flow operators such as If and Loop.
455  // The meaning of a block is specific to the kind of node it is in, but
456  // all blocks share these semantics:
457  // * Nested lexical scoping: If a node 'Parent' has a subblock which contains
458  // a node 'Child', Child can use any value that was in scope for the Parent
459  // node in addition to any values defined before 'Child' in the subblock.
460  // * The list of inputs to the block are in scope for the duration of the
461  // block
462  // * the outputs of the Parent node are not in scope for the subblocks
463  // Typically the inputs to a block that represents control flow act as
464  // as the equivalents phi-nodes in standard SSA form,
465  // defining a new Value to represent any term that has multiple
466  // definitions depending on how control flowed. Outputs of the node containing
467  // control flow serve a similiar purpose defining new values for variables
468  // that would have different defintions depending on which way control flowed.
