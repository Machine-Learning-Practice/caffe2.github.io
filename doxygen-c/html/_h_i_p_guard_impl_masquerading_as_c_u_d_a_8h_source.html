20 // Note [Masquerading as CUDA]
21 // ~~~~~~~~~~~~~~~~~~~~~~~~~~~
22 // c10_hip is very easy to understand: it is HIPified from c10_cuda,
23 // and anywhere you said CUDA, the source code now says HIP. HIPified
24 // PyTorch is much harder to understand: it is HIPified from regular
25 // PyTorch, yes, but NO source-to-source translation from CUDA to
26 // HIP occurs; instead, anywhere we see "CUDA", it actually means "HIP".
27 // For example, when you use HIPified PyTorch, you say x.cuda() to
28 // move a tensor onto ROCm device. We call this situation "HIP
29 // maquerading as CUDA".
30 //
31 // This leads to a very awkward situation when we want to call c10_hip
32 // code from PyTorch, since c10_hip is expecting things to be called
33 // HIP, but PyTorch is calling them CUDA (masquerading as HIP). To
34 // fix this impedance mismatch, we have MasqueradingAsCUDA variants
35 // for all c10_hip classes. These translate between the "HIP" and "CUDA
36 // masquerading as HIP" worlds. For example,
37 // HIPGuardImplMasqueradingAsCUDA (this file) provides something like a
38 // HIPGuardImpl, but it reports its DeviceType as CUDA (e.g., type()
39 // returns CUDA, getDevice() reports the current HIP device as a CUDA
40 // device.)
41 //
42 // We should be able to delete all of these classes entirely once
43 // we switch PyTorch to calling a HIP a HIP.
44 //
45 // When you add a new MasqueradingAsCUDA class/function, you need to
46 // also update the rewrite rules in tools/amd_build/pyHIPIFY/cuda_to_hip_mappings.py
47 //
48 //
49 //
50 // By the way, note that the cpp file associated with this also
51 // *overwrites* the entry in the DeviceGuardImpl registry for CUDA with
52 // this HIP implementation.
53
